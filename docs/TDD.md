# TDD 流程规范

> 测试驱动开发（Test-Driven Development）是本项目的核心开发方法论。
> 所有新功能和 Bug 修复都应遵循本规范。

---

## 核心铁律：RED → GREEN → REFACTOR

```
  ┌─────────┐     ┌─────────┐     ┌───────────┐
  │  RED    │────▶│  GREEN  │────▶│ REFACTOR  │
  │ 写失败  │     │ 最小实现 │     │ 清理代码  │
  │ 测试    │     │ 让测试过 │     │ 保持绿色  │
  └─────────┘     └─────────┘     └───────────┘
       ▲                                │
       └────────────────────────────────┘
                  下一个测试
```

### RED（红灯）

1. 根据需求写一个**会失败**的测试
2. **运行测试，确认它确实失败**（不运行就写实现是违规）
3. 失败原因必须是"功能未实现"，而非语法错误或配置问题

### GREEN（绿灯）

1. 写**最小量的生产代码**让测试通过
2. 不要提前优化，不要写"顺便"的代码
3. **运行测试，确认通过**

### REFACTOR（重构）

1. 在测试全绿的保护下，清理代码
2. 消除重复、改善命名、提取函数
3. **重构后再次运行测试，确认仍然全绿**
4. 重构不改变外部行为，只改善内部结构

## 测试优先级

### 新功能开发

```
PRD 审批 → 拆分 TODO → 对每个 TODO：
  1. 先写测试（RED）
  2. 验证失败
  3. 最小实现（GREEN）
  4. 验证通过
  5. 重构（REFACTOR）
  6. 验证通过
  7. 提交
```

### Bug 修复

```
Bug 报告 → 复现：
  1. 先写测试复现 Bug（RED）— 测试应在 Bug 存在时失败
  2. 验证测试确实失败（确认复现）
  3. 修复 Bug（GREEN）
  4. 验证测试通过（确认修复）
  5. 提交
```

## 测试分类

### 单元测试（Unit）

- **范围**: 单个函数 / 方法 / 类
- **特点**: 快速、隔离、无外部依赖
- **命名**: `test_<被测行为>_<场景>_<预期结果>`
- **原则**:
  - 每个测试只验证一个行为
  - 使用 mock/stub 隔离外部依赖
  - 执行时间 < 100ms

### 集成测试（Integration）

- **范围**: 多个模块协作、数据库交互、API 调用
- **特点**: 验证模块间的真实交互
- **原则**:
  - 使用测试数据库 / 测试环境
  - 测试后清理测试数据
  - 执行时间 < 5s

### 端到端测试（E2E）

- **范围**: 完整用户流程
- **特点**: 模拟真实用户操作
- **原则**:
  - 只覆盖关键业务流程
  - 保持稳定性，避免 flaky test
  - 执行时间 < 30s

## 测试编写规范

### 结构：Arrange → Act → Assert

```
def test_example():
    # Arrange — 准备测试数据和环境
    input_data = ...

    # Act — 执行被测行为
    result = function_under_test(input_data)

    # Assert — 验证结果
    assert result == expected
```

### 命名规范

- 测试文件: `test_<模块名>.py` / `<模块名>.test.ts`
- 测试函数: `test_<行为>_<场景>_<预期>` 或 `describe/it` 风格
- 测试应能作为文档阅读，名字要说清楚测试了什么

### 测试数据

- 使用 factory / fixture 生成测试数据
- 不依赖生产数据
- 测试数据应当最小化且有意义

## 反模式清单（禁止）

| 反模式 | 说明 | 正确做法 |
|--------|------|----------|
| 测试 mock 的行为 | 验证 mock 被调用了而非验证业务结果 | 验证最终输出或副作用 |
| 生产代码加测试专用方法 | 为了测试而暴露内部实现 | 通过公开接口测试，或重构设计 |
| 先写实现再补测试 | 违反 TDD 核心原则 | 严格遵循 RED-GREEN-REFACTOR |
| 测试间共享可变状态 | 测试顺序依赖，导致 flaky | 每个测试独立 setup/teardown |
| 过度 mock | mock 了大半个系统，测试无意义 | 只 mock 外部依赖（网络、文件系统、时间） |
| 忽略失败测试 | 用 skip/ignore 掩盖问题 | 修复或删除过时测试 |
| 测试实现细节 | 重构就破测试 | 测试行为和接口，不测内部实现 |

## 与项目工作流的集成

### 循环模式下的 TDD

在 `claude -p` 循环模式中：

1. TODO 中标注需要测试的任务
2. Agent 执行时先写测试，再写实现
3. `completed=true` 的前提包括：测试存在且通过

### 交互模式下的 TDD

在 Claude Code 交互模式中：

1. 用户提出需求后，Agent 先建议测试用例
2. 确认后先写测试，运行验证失败
3. 再写实现，运行验证通过
4. 重构后再次验证
